这里我先简单介绍一下status这个文件

CPU Affinity API

cpu_set_t 数据结构

cpu_set_t 是一个位图结构，每一位代表一个 CPU 核心

如果某位为 1，表示该 CPU 在集合中；为 0 表示不在集合中

CPU_ZERO(&c)

将 CPU 集合中的所有位清零

清空后，集合中不包含任何 CPU 核心

相当于创建一个空集合

CPU_SET(i, &c)

将第 i 个 CPU 核心添加到集合中

实际上是将位图中第 i 位设置为 1

i 从 0 开始计数（0 表示第一个核心）
cpu_set_t 本身只是一个数据结构（位图），它可以用来表示任意的 CPU 集合。真正决定它作用于哪个进程的是 sched_setaffinity() 函数的第一个参数：

check_crash_handling

这个函数的核心目的是确保系统的崩溃处理机制不会干扰 AFL 的正常运行，因为外部崩溃处理程序会导致检测延迟，影响 fuzzing 效率。

Linux就是检查了下 core_pattern 的第一个字符如果第一个字符是 ‘|’（管道符），表示核心转储会被发送到外部程序处理，为什么要这样做哪？有以下几个原因：

性能影响：外部处理程序可能需要几秒钟来处理崩溃，严重降低 fuzzing 速度

准确性影响：延迟可能导致 AFL 误判（崩溃 → 超时）

资源消耗：外部崩溃报告工具会消耗额外的 CPU 和内存

check_cpu_governor

这个函数是用来检查 CPU 频率调节策略（CPU Scaling Governor），确保 CPU 运行在最佳性能模式，避免动态频率调节影响 fuzzing 性能。

setup_post

核心目的

设置 AFL 的后处理器（postprocessor），允许用户提供自定义的测试用例后处理逻辑。

后处理器的作用

AFL 后处理器是一个可选的自定义函数，在 AFL 生成测试用例后、发送给目标程序前被调用：

AFL 生成测试用例 → 后处理器修改 → 发送给目标程序

常见用途：

  1. 格式转换：将 AFL 的原始数据转换为特定格式
  2. 校验和修复：修复数据的校验和、CRC等
  3. 协议封装：添加协议头部、长度字段等
  4. 数据压缩：对数据进行压缩或编码

函数接口

后处理器必须实现以下接口：

参数：

  • buf：输入/输出缓冲区，包含要处理的数据
  • len：指向长度的指针，输入时是原始长度，输出时是处理后的长度

也就是这个函数是提供我们一个接口从而修改我们AFL生成的test case，以针对不同的情况

mark_as_det_done

这里先提一下确定性变异的概念

确定性变异（Deterministic Fuzzing）说明

在AFL中，对每个测试用例的变异分为两个阶段：

  1. 确定性阶段

    （Deterministic）：

    • 系统地尝试所有预定义的变异操作
    • 包括位翻转、算术运算、插入特殊值等
    • 每个操作都会执行，不随机
    • 耗时较长但覆盖全面

  2. 随机性阶段

    （Havoc）：

    • 随机组合各种变异操作
    • 更快但可能遗漏某些路径

标准输出和标准错误的重定向
  • 功能：将标准输出（文件描述符 1）和标准错误（文件描述符 2）重定向到 /dev/null
  • 目的：
    • 确保守护进程的任何输出都不会显示在终端上
    • 防止向可能已关闭的终端写入数据导致进程收到 SIGPIPE 信号而崩溃

标准输入的重定向
  • 功能：根据是否指定了输出文件，选择不同的方式重定向标准输入（文件描述符 0）
  • 目的：
    • 如果指定了输出文件（out_file），将标准输入重定向到 /dev/null，表示进程不需要任何输入
    • 如果没有指定输出文件，将标准输入重定向到 out_fd(setup_stdio_file中赋值的)，然后关闭原始的 out_fd
控制管道和状态管道的重定向
  • 功能：将控制管道的读端和状态管道的写端重定向到预定义的文件描述符位置，也就是子进程中会写死一个FORKSRV_FD来和父进程进行通信，读取命令或者返回状态
  • 目的：
    • 将控制管道的读端（ctl_pipe[0]）复制到固定的文件描述符 FORKSRV_FD
    • 将状态管道的写端（st_pipe[1]）复制到固定的文件描述符 FORKSRV_FD + 1
    • 这样做是为了让后续通过 execv 执行的子进程能够通过固定的文件描述符与父进程通信，而不需要知道原始的管道文件描述符

看完这个函数我们看看函数所在的循环，循环会执行 stage_max 次（默认初始为8次），每次迭代中先将测试用例数据写入目标程序输入文件，然后运行目标程序并监控其执行状态（如超时或崩溃）；首次运行时记录初始路径覆盖轨迹（trace_bits）的校验和，后续运行会对比校验和是否变化——若变化则说明执行路径发生波动，此时会标记出具体变化的字节位并延长校准轮次（从8次增至40次），同时更新“新路径覆盖”状态；如果期间遇到外部中断信号（如Ctrl+C）或执行故障（如目标程序崩溃），则提前终止校准。最终，校准结果用于判断该测试用例的可靠性（是否适合用于后续变异）以及记录其初始路径覆盖信息。在第一次循环结束会把该次trace_bits赋值到**first_trace**，以便这个循环后面使用，而且会把第一次测试的校验和加入到q->exec_cksum，以便和后面对比，做一个快速筛。至于循环每次的信息会报存在var_bytes中

var_bytes 数组的作用：它的每一位（bit）标志着对应位置的路径分支（即 first_trace 中的那个字节）在多次校准执行中是否曾经发生过变化。

  • var_bytes[i] = 0：表示在第 i 个位置上的计数在整个校准过程中始终稳定。
  • var_bytes[i] = 1：表示这个位置至少在一次执行中与第一次记录的值不同，它是一个不稳定的“波动位”。

common_fuzz_stuff

先检查用户是否设置了自定义的后处理函数，如果有则调用，然后把程序写入临时文件，既out_file。后面比较重要的就是save_if_interesting函数，我们详细看看

save_if_interesting

如果没有新的路径则返回，如果有新的路径或者命中次数变化就将其命名并添加到fuzzing队列，如果有新的路径则将其标记有新覆盖。

然后记录路径校验和并通过calibrate_case多次检验记录信息，然后再写入到磁盘当中.然后处理特殊的fault。

最后返回keeping，只要样例写入磁盘保存到queue/目录keeping就为1，这样这个样例会作为种子被后续变异，返回common_fuzz_stuff后会使得queued_discovered自增。

然后就到了AFL的自动字典提取机制，这里先是只在每个字节的最后一位触发，这部分有个a_collect数组，数组的作用是记录可能的token，如果是文件末尾，且这次校验和等于上次校验和则直接加入a_collect，防止遗漏，然后就会检查是否长度合理，如果合理就调用maybe_add_auto函数，另一种情况就是检测到可能是token的边界情况，比如说进入token区域或者走出token区域，就是这次校验和和上次不一样的情况，然后就判断长度是否合理再调用maybe_add_auto函数来保存。之后就要将len置为0，因为这里可能是这个token的开始，之后要开始记录，这里清理数组，也可能是之前token已经加入，这里也得群里准备之后的token。

接下来就收集活跃字节，这里是前面文件末尾添加的普遍情况，如果校验和和之前没有变异的情况不一样的话就说明是活跃字节，就加入。

我们看看这个maybe_add_auto函数具体做了什么

maybe_add_auto

这个函数先检查是否所有字节是否全部相同，如果全部相同则不保存，然后根据token的长度不同来考虑是否和已知有趣值来比较包括大端和小端模式，然后和用户字典条目比较，同样不能相同，最后检查a_extras也就是自动字典，如果也没有重复就说明完全没有重复，这样就将其加入到a_extras

在此之后我们可以看到一个Effector Map

首先是EFF_APOS(_p)这个比较好理解，就是对_p进行除以8，

EFF_REM(_x)是计算偏移在效应图条目中对*(1 << EFF_MAP_SCALE2)*取模，

EFF_ALEN(_l)是_l除以8然后向上取整，

!!EFF_REM(_l)可以详细聊聊，这里!!是一个双重取反实际上是将任意值转换为标准布尔值（0或1），如果为0就是0，否则就是1。

EFF_SPAN_ALEN(_p, _l) ：

**EFF_APOS(_p)**：起始偏移 _p 所在的条目索引

**EFF_APOS((_p) + (_l) - 1)**：最后一个字节所在的条目索引

  • (_p) + (_l) - 1 是最后一个字节的偏移

索引差 + 1：得到跨越的条目数量

算数运算阶段

首先是8位运算阶段，先检查是否对加法和减法进行测试，对一个字节的内容进行操作，从+1到+35，从-1到-35，对该字节进行操作后先检测是否和之前位取反的操作重复，如果重复就跳过，这样可以加快fuzz的速度，提高效率，具体是怎么判断的我们看看could_be_bitflip

could_be_bitflip

也就是和之前的各种翻转结果去对比，如果相同就返回1。

接下来是16位运算阶段，16位和8位有一些区别，16位存在大小端的区别，会对大端和小端分别测试，其他和8位差不多，只不过是只有影响到两个字节的时候才可能调用common_fuzz_stuff。

同样的32位也是4种，有大小端的区别，只有影响超过3个字节才会调用common_fuzz_stuff。
